# 咩了个咩 · 系统架构

> 这份文档是项目的"鸟瞰图"。
> 看完你应该能回答三个问题：
> **文件夹怎么分？模块怎么互相调用？玩家点一下牌后数据怎么流？**

---

## 一、整体形态（最高视角）

我们做的是一个**纯前端静态网站**：

```
玩家浏览器
    ↓ 访问 https://你的用户名.github.io/sheep-stack/
GitHub Pages 服务器
    ↓ 返回静态文件 (HTML / CSS / JS / 图片 / 音频)
玩家浏览器
    ↓ 浏览器解析并执行
游戏在浏览器里跑（无需服务器）
```

**没有数据库**、**没有后端 API**、**没有账号系统**。所有数据要么在内存里，要么在玩家自己的浏览器 `localStorage` 里。

```
├── LICENSE                     # 开源协议（建议 MIT）
├── progress.md                 # 短期记忆，开发进度
├── .gitignore                  # 告诉 Git 哪些文件不上传
│

│   ├── 04-开发计划.md
│   └── decisions/              # 重要决策记录（ADR）

├── src/                        # 源码目录
│   ├── main.js                 # 程序入口（被 HTML 第一个加载的 JS）
│   ├── config.js               # 全局常量配置（牌数、槽位数等）
│   ├── state.js                # 全局状态对象 + 修改方法
│   │
│   ├── game/                   # 游戏核心逻辑（不碰 DOM）
│   │   ├── tile.js             # 单张牌的定义、生成
│   │   ├── board.js            # 牌堆生成、点击判定
│   │   ├── slot.js             # 底部槽位逻辑、消除判定
│   │   ├── level.js            # 关卡数据加载
│   │   └── rules.js            # 胜负判定
│   │
│   ├── ui/                     # 界面层（只碰 DOM）
│   │   ├── render.js           # 把 state 渲染到屏幕

│   │
│   ├── audio/
│   │   └── sound.js            # 音效播放
│   │
│   └── storage/

│
├── assets/                     # 静态资源
│   ├── images/                 # 牌面图标、按钮
│   ├── sounds/                 # 音效
│   └── fonts/                  # 字体（如有）
│
├── styles/                     # 样式
│   ├── base.css                # 重置、CSS 变量
│   ├── layout.css              # 布局
│   └── theme.css               # 颜色、皮肤
│
└── levels/                     # 关卡数据（JSON 文件）
    ├── level-01.json
    ├── level-02.json
    └── ...
```

**为什么这么分？（小白版）**

- `game/`（**纯逻辑**） vs `ui/`（**碰 DOM**）严格分开 → 将来想换成 React 或 Canvas 渲染，`game/` 可以直接复用，不用重写。
- `levels/` 用 JSON 单独存 → 改关卡难度不用改代码，改 JSON 就行。
- `assets/` 集中放 → 将来要换皮肤、换音效，目录清晰。
- `styles/` 拆成三块 → `base` 是地基、`layout` 是骨架、`theme` 是皮肤。换主题只动 `theme.css`。

## 三、模块依赖关系

```
                    ┌──────────────┐
                    │  index.html  │
                    └──────┬───────┘
                           ↓
                    ┌──────────────┐
                    │   main.js    │  ← 入口
                    └──┬────────┬──┘
                       ↓        ↓
              ┌────────────┐  ┌──────────────┐
              │  state.js  │  │  config.js   │
              └─────┬──────┘  └──────────────┘
                    ↓
        ┌───────────┼───────────┐
        ↓           ↓           ↓
   ┌────────┐  ┌────────┐  ┌──────────┐
   │ game/  │  │  ui/   │  │ storage/ │
   │ (纯JS) │  │ (DOM)  │  │ (localS) │
   └────────┘  └────────┘  └──────────┘
                    ↓
              ┌──────────┐
              │ audio/   │
              └──────────┘
```

**调用规矩**（必须遵守，否则项目会乱成一锅粥）：

- `ui/` 可以调 `game/`，但 `game/` **不能调** `ui/`。
- `game/` **不能直接读写** `localStorage`，要走 `storage/`。
- 所有模块都通过 `state.js` 拿 / 改全局状态。

## 四、核心数据结构

### 4.1 一张牌（Tile）

```js
{
  id: 'tile-42',           // 唯一 ID
  type: 'sheep',           // 图案类型，比如 'sheep' / 'grass' / 'wolf'
  layer: 2,                // 在牌堆第几层（0 是最底）
  x: 100, y: 80,           // 在屏幕上的坐标（像素）
  coveredBy: ['tile-58'],  // 被哪些牌压着（这些都消失它才能点）
  status: 'onBoard'        // onBoard | inSlot | removed
}
```

### 4.2 全局状态（state）

```js
{
  level: 1,                // 当前关卡
  tiles: [ /* 所有牌的数组 */ ],
  slot: [ /* 槽位里的牌 ID 列表，最多 7 个 */ ],
  score: 0,
  status: 'playing',       // playing | won | lost | paused
  history: [ /* 撤销栈：上一步把哪张牌移到槽里了 */ ]
}
```

### 4.3 关卡 JSON

```json
{
  "id": 1,
  "name": "新手村",
  "tileTypes": ["sheep", "grass", "wolf"],
  "tileCountPerType": 6,
  "layers": 3,
  "slotCapacity": 7
}
```

**说明**：每种图案出现 `tileCountPerType` 张，必须是 **3 的倍数**（不然永远消不完）。

## 五、数据流（玩家点了一张牌后发生啥）

```
1. 玩家点击屏幕上的牌
   ↓
2. ui/controls.js 捕获 click 事件
   ↓
3. 取到被点牌的 ID，调用 game/board.js → canPickTile(id)
   ↓
4. ┌─ 不能点（被压着）→ 抖一下提示
   └─ 能点 → 调用 game/slot.js → moveTileToSlot(id)
   ↓
5. state 被更新（tile.status = 'inSlot'，slot 数组 push）
   ↓
6. game/slot.js → tryEliminate()  检查槽里有没有 3 个一样的
   ┌─ 有 → 移除三张，加分
   └─ 没 → 检查槽是不是满了
   ↓
7. game/rules.js → checkGameOver()
   ┌─ 牌堆空了 → status = 'won'
   ├─ 槽满 7 张 → status = 'lost'
   └─ 都不是 → 继续玩
   ↓
8. ui/render.js 监听 state 变化 → 重画屏幕
   ↓
9. ui/effects.js 触发动画、audio/sound.js 播音效
   ↓
10. 如果赢/输 → storage/localStorage.js 保存进度
```

**单向数据流**：永远是 `事件 → 改 state → 重渲染`。
**不允许**直接改 DOM 跳过 state（不然 state 和屏幕显示就对不上了）。

## 六、关键设计取舍

| 决定 | 选了 | 没选 | 理由 |
|---|---|---|---|
| 渲染方式 | DOM + CSS | Canvas | 牌数有限（百级），DOM 够用且方便加 CSS 动画 |
| 状态管理 | 单一 state 对象 | Redux / Zustand | v1 体量小，没必要 |
| 模块化 | ES Modules | CommonJS / 全局变量 | 浏览器原生支持，零配置 |
| 关卡格式 | JSON 文件 | 写死在代码里 | 改关卡不改代码 |
| 持久化 | localStorage | IndexedDB / 后端 | 数据量小，无需账号 |
| CSS 组织 | 原生 CSS + 变量 | SCSS / Tailwind | 减少工具链 |

## 七、未来演进路径（先记下，不立刻做）

- **v2**：引入 Vite，加热更新 / 模块打包。
