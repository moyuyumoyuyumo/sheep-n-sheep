# 咩了个咩 · 开发宪法

> 「宪法」= 项目里**绝对不能违反**的几条规矩。所有后续决策都不能和它打架。
> 如果哪天发现确实需要打破宪法，必须先把宪法**修订一次**，再去做事，而不是悄悄绕过。

---

## 一、项目定位

- **类型**：网页版三消益智小游戏，玩法借鉴《羊了个羊》。
- **目的**：个人学习、技术练手、可放进简历/作品集。
- **用户**：用浏览器（手机或电脑）直接玩的普通玩家。
- **非目标**：不做付费、不做服务器联机、不做账号系统（至少 v1 不做）。

## 二、不可违反的红线（硬约束）

| 红线 | 说明 |
|---|---|
| **不商用** | 仅供个人 / 教学 / 作品集展示，README 里必须写明。 |
| **不抄素材** | 图标、音效、字体一律使用开源资源或自制，不直接搬运原版。 |
| **不抄名字** | 项目名 ≠「羊了个羊」。可叫「咩了个咩」「叠叠羊」「Sheep Stack」等。 |
| **不存私密信息** | 不收集玩家任何个人信息（哪怕将来加排行榜也只存昵称）。 |
| **不锁死端** | 必须同时支持手机和电脑浏览器，不允许只在电脑能玩。 |

## 三、技术选型原则（软约束）

按"先简后繁"的顺序：

- **第一性原理**：能用更简单的方案解决，就**不要**上复杂的。
- **零后端起步**：v1 完全是静态页面，部署到 GitHub Pages 即可。
- **零构建起步**：v1 不引入 Vite / Webpack，直接 HTML+CSS+JS，浏览器打开就能跑。
- **零框架起步**：v1 不引入 React / Vue。先搞懂"原生怎么写"，等业务复杂再迁移。
- **演进路径**（仅在出现具体痛点时再升级）：
  1. v1：HTML + CSS + 原生 JavaScript（ES Modules）
  2. v2：必要时引入 Vite（提升开发体验）
  3. v3：必要时引入 TypeScript / 框架

## 四、技术栈（v1 锁定）

- **HTML5**：语义化结构。
- **CSS3**：布局动画用 Flex / Grid + transform；不使用 SCSS / PostCSS。
- **JavaScript (ES2020+)**：用 ES Modules（`<script type="module">`），不用打包器。
- **存储**：浏览器 `localStorage` 保存进度 / 最高分。
- **托管**：GitHub Pages（免费、永久、HTTPS）。
- **版本控制**：Git + GitHub。

## 五、性能与体验底线

- 首屏加载 ≤ **2 秒**（在中等网络下）。
- 操作响应 ≤ **100 毫秒**（点一下牌马上有反应）。
- 帧率目标 **60fps**，最低不低于 30fps。
- 在 iPhone SE 这种 320px 宽的小屏上不能错位。

## 六、代码风格

- **命名**：见名知意，宁可长不要短。`tile`、`slot`、`board` 这种术语统一。
- **注释**：用**中文**。每个函数顶上一句话说"这函数干嘛"。
- **函数**：单个函数 ≤ 50 行，超过就拆。
- **文件**：单个 JS 文件 ≤ 300 行，超过就拆模块。
- **不留死代码**：注释掉的代码必须删掉，靠 Git 历史回找。
- **不用魔法数字**：`7`、`3` 这种关键数字写成命名常量（如 `SLOT_CAPACITY = 7`）。

## 七、协作约定（你和我之间）

- **沟通语言**：全程中文。
- **解释优先**：遇到新概念，先解释"是什么、为啥要它"，再写代码。
- **小步快跑**：每次只推进一小步，跑通了再做下一步，不憋大招。
- **可回滚**：每个阶段做完都提交一次 Git，万一搞砸能退回。
- **文档先行**：动手前先有"这一步要做什么"的清单，做完再回头打勾。

## 八、修订规则

这份宪法不是写死的，但每次改动要：
1. 在文档底部加一条 **修订记录**（日期 + 改了啥 + 为啥改）。
2. 在 Git 提交里说明原因。

不要悄悄改。
