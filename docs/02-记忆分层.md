# 咩了个咩 · 记忆分层

> 「记忆分层」= 把项目里**所有"需要被记住的东西"**按"活多久"分层管理。
> 这样以后改东西的时候，你能立刻判断："这个信息该改哪里"。

---

## 为啥要分层？（小白版）

举个例子：

- 你的**身份证号** —— 一辈子的事 → 写进户口本。
- 你今年的**工作目标** —— 一年有效 → 写进工作笔记。
- 你今天**要买的菜** —— 半天有效 → 记在便利贴上。
- 你刚刚**算到一半的算式** —— 几秒钟 → 记在脑子里。

软件项目里的"信息"也一样。胡乱混在一起会出大问题：

- 你想改"游戏一关有几张牌" → 不知道改哪里。
- 你想看"为啥当初选 localStorage" → 没记录，忘了。
- 你想知道"昨天写到哪了" → 全靠脑子记。

下面把咱项目的"记忆"分成 6 层。

---

## L1 · 长期记忆（项目宪法 / 架构）

- **位置**：`docs/01-开发宪法.md`、`docs/03-系统架构.md`
- **寿命**：项目全周期（除非重写）
- **改动频率**：极低
- **内容**：
  - 项目使命、红线、技术栈选择
  - 模块划分、数据流、文件结构

**判断标准**：如果改动它，相当于"项目调头"，必须慎重。

## L2 · 中期记忆（计划 / 决策记录）

- **位置**：`docs/04-开发计划.md`、`docs/decisions/`（ADR）
- **寿命**：以"阶段"为单位（数周到数月）
- **改动频率**：每阶段更新
- **内容**：
  - 当前阶段目标、做完的标志
  - "为什么 v1 不用 React" 这种重大决定的来龙去脉

**ADR 是什么？** = Architecture Decision Record，**架构决策记录**。每做一个重要决定就写一个 md 文件存档，将来回头看才知道"当初为什么这么选"。

**ADR 模板**（`docs/decisions/0001-为什么不用React.md`）：

```markdown
# 0001 · 为什么 v1 不用 React

## 背景
v1 阶段就一个游戏页面，组件不超过 10 个。

## 选项
- A：直接 DOM 操作（原生 JS）
- B：React
- C：Vue

## 决定
选 A。

## 后果
- 好：零依赖、零构建、新人 3 分钟看懂。
- 坏：状态多了写起来啰嗦，超过 30 个组件时考虑迁 B。
```

## L3 · 短期记忆（进度日志 / TODO）

- **位置**：`progress.md`（项目根目录）
- **寿命**：一次开发会话到一两周
- **改动频率**：几乎每次写代码都改
- **内容**：
  - 这次会话做到哪了
  - 下次接着做啥
  - 临时遇到的小问题

**用法**：每次和我（Cascade）开新会话时，让我先看一眼 `progress.md`，我立刻知道上下文。

## L4 · 代码内记忆（注释 / 命名）

- **位置**：源码里
- **寿命**：和代码一样长
- **改动频率**：跟着代码改
- **内容**：
  - 函数顶部一句话："这函数干啥"
  - 复杂逻辑旁边的"为什么这么写"
  - 有意义的变量名（变量名本身就是注释）

**反例**：

```js
let a = 7;          // 不好：a 是啥？
let n = arr.length; // 不好：n 是啥？

const SLOT_CAPACITY = 7;  // 好：一看就懂
const tileCount = arr.length; // 好
```

## L5 · 运行时记忆（游戏内存）

- **位置**：JavaScript 运行时的全局 `state` 对象
- **寿命**：从打开网页到关闭网页
- **改动频率**：每次玩家点击都可能变
- **内容**：
  - 当前关卡 ID
  - 牌堆里还剩哪些牌
  - 槽位里现在有啥
  - 当前得分、剩余时间、是否暂停

**关键点**：所有游戏状态收拢到**一个 `state` 对象**里，方便调试（在浏览器控制台 `console.log(state)` 就能看全局）。

## L6 · 持久化记忆（玩家存档）

- **位置**：浏览器 `localStorage`（v1 阶段）
- **寿命**：除非用户清浏览器，否则永久
- **改动频率**：通关、得分变化时
- **内容**：
  - 已通关的最高关卡
  - 历史最高分
  - 玩家自定义设置（音效开关、皮肤）

**数据格式**（统一用 JSON）：

```json
{
  "version": 1,
  "highestLevel": 3,
  "highScore": 1280,
  "settings": { "sound": true }
}
```

**注意**：`version` 字段必填，将来改格式时方便迁移老存档（不然老玩家进来会报错）。

---

## 总览图

| 层 | 在哪 | 活多久 | 谁改 |
|---|---|---|---|
| L1 长期 | `docs/01`, `docs/03` | 项目一辈子 | 你（重大决定时） |
| L2 中期 | `docs/04`, `docs/decisions/` | 几个阶段 | 你（每阶段） |
| L3 短期 | `progress.md` | 几天 | 你 + 我（每次会话） |
| L4 代码内 | 源码注释 | 跟代码 | 我写代码时 |
| L5 运行时 | `state` 对象 | 一次会话 | 游戏代码 |
| L6 持久化 | `localStorage` | 除非清浏览器 | 游戏代码 |

---

## 维护规则

- **L1 / L2** 改动 → 必须在文档底部留"修订记录"。
- **L3** 改动 → 直接覆盖旧内容即可，不必留痕。
- **L5** 数据 → 永远只通过函数读写，不允许散落赋值。
- **L6** 数据 → 改格式时必须升 `version` 字段并写迁移函数。
